React 톺아보기- 03. Hooks_1

12 days ago

모든 설명은 v16.12.0 버전 함수형 컴포넌트 기준입니다. 버전에 따라 코드는 변경될 수 있으며 클래스 컴포넌트는 설명에서 제외됨을 알려 드립니다.

글을 써내려가는 방식은 하나도 몰랐던 상태에서 리액트를 분석한 그 순서 그대로를 따라가면서 설명하도록 하겠습니다.
글의 흐름은 다음과 같습니다.

  1. 훅을 통해 컴포넌트 상태를 업데이트한다.
  2. VDOM 재조정을 해줄 Work를 scheduler에게 스케줄링 시키고 scheduler가 스케줄링된 Task를 꺼내 실행한다
  3. Work을 통해 VDOM 재조정 작업을 진행한다.
  4. 완성된 VDOM을 commit phase에서 처리하여 DOM에 적용한다.
  5. 사용자의 상호작용으로 이벤트가 발생하고 해당 이벤트에 등록된 핸들러가 실행되면서 다시 1번으로 되돌아간다.

위 내용이 리액트에서 가장 일반적인 흐름일 것으로 생각됩니다. 위 내용만 제대로 이해하게 된다면 이 이외의 케이스들에 대해서 굳이 설명하지 않아도 혼자서 충분히 알아볼 수 있지 않을까 기대해봅니다.

이번 포스트에서는 첫 번째인 훅을 통해 컴포넌트 상태를 어떻게 업데이트하는지 알아보도록 하겠습니다.

1. Hook의 구현체는 어디에 있을까?

분석을 시작하기 가장 좋은 방법은 분석할 함수를 어디서 어떻게 가져오는지 먼저 확인하는 것입니다.

// react > React.js
import { useState, useEffect } from './ReactHooks'
import ReactSharedInternals from './ReactSharedInternals'

const React = {
  // 생략..
  useState,
  useEffect,
  // 생략..
  __SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED: ReactSharedInternals,
}

export default React

개발자가 코어를 통해 가져오는 훅은 ReactHooks 모듈에서 가져오고 있습니다.

// react > ReactHooks.js
import ReactCurrentDispatcher from './ReactCurrentDispatcher'

export function useState(initialState) {
  const dispatcher = resolveDispatcher()
  return dispatcher.useState(initialState)
}

export function useEffect(create, inputs) {
  const dispatcher = resolveDispatcher()
  return dispatcher.useEffect(create, inputs)
}

function resolveDispatcher() {
  const dispatcher = ReactCurrentDispatcher.current
  return dispatcher
}

ReactCurrentDispatcher란 모듈에서 dispatcher를 가지고 오네요. 훅의 구현체는 dispatcher가 가지고 있나 봅니다.

// react > ReactCurrentDispatcher.js
const ReactCurrentDispatcher = {
  current: null,
}

export default ReactCurrentDispatcher

아무것도 없습니다.. 그냥 객체 하나가 끝입니다.

preview에서 패키지를 설명할 때 코어는 “개발자에게 다른 패키지의 모듈을 제공할 수 있도록 중간 다리 역할을 하는 모듈들만 위치해 있습니다.” 라고 말했었습니다.

훅에 대해서 좀만 더 생각해보자면 훅은 컴포넌트의 상태를 업데이트하는데 코어에서는 컴포넌트의 겉 정보인 reactElement만을 관리합니다.
그렇다는 이야기는 이 상태는 VDOM의 fiber에 저장된다는 것이고 이 fiber를 업데이트할 수 있는 역할은 fiber를 만들어내는 reconciler에게 있다는 말입니다. 그러므로 hook의 구현체 또한 reconciler에 위치해 있습니다.

이렇게 보면 코어가 다른 패키지의 기능을 개발자에게 제공해 줄 때 의존성을 자기가 만들지 않고 외부에서 주입 받는다는 걸 알 수 있습니다. 스프링의 DI(Dependency Injection)와 같습니다.
리액트는 외부에서 의존성을 주입할 때 코어에 바로 주입하지 않습니다. 중간자를 하나 더 두게 되는데 그게 Shared라는 패키지입니다.
이 Shared는 모든 패키지들이 공유하는 정보들을 가지고 있습니다. intro에서 본 side effect tag들도 여기에 위치해 있습니다.

Shared에서 코어에 의존성을 주입해주기 위해 react를 알고 있는 모듈은 ReactSharedInternal.js입니다.

// shared > ReactSharedInternals.js
import React from 'react'

const ReactSharedInternals =
  React.__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED

if (!ReactSharedInternals.hasOwnProperty('ReactCurrentDispatcher')) {
  ReactSharedInternals.ReactCurrentDispatcher = {
    current: null,
  }
}
// 생략..

export default ReactSharedInternals

맨 처음 React 선언 코드에서 본 __SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED를 여기에서 사용하고 있습니다. 여기에는 코어의 ReactSharedInternals 이 할당되어 있었습니다.

// react > ReactSharedInternals.js
import ReactCurrentDispatcher from './ReactCurrentDispatcher'

const ReactSharedInternals = {
  ReactCurrentDispatcher,
  // 생략..
}

export default ReactSharedInternals

훅이 개발자에게 도달되는 흐름은 다음과 같습니다. reconciler -> shared/ReactSharedInternal -> react/ReactSharedInternal -> react/ReactCurrentDispatcher -> react/ReactHooks -> react -> 개발자의 형태가 됩니다. 다시 코드를 한 번 쭉 훑어 보시면서 흐름을 느껴 보시길 바랍니다.

이제 우리는 훅이 어디에 있는지 shared/ReactSharedInternal.js를 import하고 ReactCurrentDispatcher를 사용하고 있는 곳을 찾아가면 확인할 수 있을 것입니다.
훅은 reconciler/ReactFiberHooks.js에 위치해 있습니다.

2. Hook은 내부에서 어떻게 사용되는가?

본격적으로 들어가기에 실제 코드를 보면 __DEV__로 감싸진 코드들을 많이 보실 텐데 개발 모드에서만 사용됨을 뜻하기 때문에 산뜻하게 무시해주시면 됩니다.

훅을 ReactCurrentDispatcher.current에 할당해주는 함수는 renderWithHooks()입니다. 함수 이름에서 느껴지시나요? 컴포넌트 실행을 여기서 합니다. 주목적은 훅 할당이 아니라 함수 실행이고 훅 할당은 부수적으로 따라오는 부분이 됩니다.
이 함수는 재조정 작업을 할 때 사용되는데 그때 가서 자세히 다루고 지금은 훅과 관련된 코드들만 뜯어와서 보도록 하겠습니다.
코드에서 제가 설명하지 않고 넘어가는 코드는 크게 신경 쓰지 않으셔도 됩니다. 복잡하니 그때는 생략되고 사용되는 뒷부분에서 다뤄진다고 생각하시면 됩니다.

분석하기에 type이 있으면 좋을 부분들은 type를 제거하지 않고 그대로 사용하도록 하겠습니다.

// reconciler > ReactFiberHooks.js > renderWithHooks()
export function renderWithHooks(
  current: Fiber,
  workInProgress: Fiber,
  Component: any,
  props: any,
  refOrContext: any,
  nextRenderExpirationTime: ExpirationTime
) {
  // 생략..
  currentlyRenderingFiber = workInProgress // 현재 작업 중인 fiber를 전역으로 잡아둠
  nextCurrentHook = current !== null ? current.memoizedState : null

  ReactCurrentDispatcher.current =
    nextCurrentHook === null ? HooksDispatcherOnMount : HooksDispatcherOnUpdate

  let children = Component(props, refOrContext)

  const renderedWork = currentlyRenderingFiber
  renderedWork.memoizedState = firstWorkInProgressHook

  ReactCurrentDispatcher.current = ContextOnlyDispatcher
  // 생략..
}

19, 22 라인을 먼저 확인하겠습니다.
Component는 fiber의 type에서 꺼낸 온 것인데 여기에는 개발자가 작성한 컴포넌트 함수가 저장되어 있습니다. 19 라인을 통해 훅을 사용하고 있는 컴포넌트를 실행하면 전역변수 firstWorkInProgressHook에 hook 객체가 저장됩니다. 이 변수를 fiber의 memoizedState에 저장해 놓음으로서 훅을 컴포넌트와 매핑시켜 줍니다.

이제 12번 라인을 이해할 수 있습니다. memoizedState가 null이 아니라면 이전 컴포넌트 렌더에서 훅이 사용되었다는 뜻이 됩니다.
그리고 15 라인에서 nextCurrentHook을 이용하여 훅이 mount인지 update인지 판단을 하게 됩니다. 그리고 거기에 맞게 훅 구현체를 다르게 사용합니다.
즉 컴포넌트가 mount 될 때 훅은 mount 용 구현체를 사용할 것이고 그 이후에는 컴포넌트가 unmount되지 않는 한 계속 update 용 구현체를 사용하게 됩니다.

마지막으로 24번 라인에서 한 번 더 다른 훅 구현체 ContextOnlyDispatcher를 사용합니다. 이는 컴포넌트 실행이 모두 끝난 다음 혹시나 훅을 호출하는 상황이 발생할 때 에러를 던져 개발자가 올바르게 훅을 사용할 수 있도록 해주는 장치입니다.

// reconciler > ReactFiberHooks.js

// mount
const HooksDispatcherOnMount = {
  useState: mountState,
  useEffect: mountEffect,
};

// update
const HooksDispatcherOnUpdate: = {
  useState: updateState,
  useEffect: updateEffect,
};

// invalid hook call
export const ContextOnlyDispatcher: Dispatcher = {
  useState: throwInvalidHookError,
  useEffect: throwInvalidHookError,
};

3. Hook은 어떻게 구현되어 있을까?

이번 포스트의 분석 대상인 훅은 useState 입니다.

분석하실 때 함수 내부에서 선언된 지역변수인지 모듈 최상단에서 선언된 전역변수인지 잘 확인하시길 바랍니다.

Mount state

1. 훅 객체 만들기

mount이기 때문에 먼저 훅 객체를 만들어 줍니다.

// reconciler > ReactFiberHooks.js > mountState()
function mountState(initialState) {
  // 훅 객체를 생성한다.
  const hook = mountWorkInProgressHook()
  // 생략..
}

ReactFiberHooks 모듈의 컨텍스트는 현재 작업 중인 컴포넌트 전용으로 사용됩니다. 이 말이 무슨 뜻이냐면 해당 모듈에서 선언되는 모든 것들은 현재 컴포넌트에만 국한되는 상태값으로 관리된다는 뜻입니다. 컴포넌트의 작업이 끝나게 되면 모두 초기화시켜 다음 컴포넌트에서 사용할 수 있도록 준비시켜 놓습니다.

mountWorkInProgressHook에서 hook객체를 만들고 컴포넌트에서 맨 처음으로 호출된 훅일 경우에 전역 변수인 firstWorkInProgressHook와 workInProgressHook에 저장해 놓습니다.
이제 변수명에서 first만 보면 바로 연결 리스트로 생각 하셔도됩니다. 리액트는 많은 곳에서 built-in collection 대신 연결 리스트를 이용하여 구현했습니다.

built-in collection이 아닌 연결리스트로 구현된 이유는 순회할 때 제어하기가 매우 쉽기 때문입니다.
순회 조건이나 중지, 노드의 삭제 등 제어가 간편합니다.
또 다른 이점 하나는 연결 리스트끼리 이어 붙이는데 많은 리소스가 들지 않는다는 겁니다. 단지 tail 포인터만 연결할 리스트의 head를 가리키기만 하면 됩니다.
더불어 리액트에서는 랜덤 엑세스가 필요한 부분이 없으므로 더욱이 연결리스트를 쓰지 않을 이유가 없어 많은 곳에서 사용되고 있습니다.

firstWorkInProgressHook은 renderWithHooks()에서 이미 봤듯이 컴포넌트 실행이 끝났을 때 renderedWork.memoizedState = firstWorkInProgressHook;으로 fiber에 저장하는데 사용되고 workInProgressHook은 연결 리스트에서 현재 처리되고 있는 hook에 대한 포인터로 사용됩니다.
여러 블로그에서 훅 구현체를 설명할 때 index와 배열을 이용하여 설명하지만 우리는 이제 실제 코드를 통해 연결 리스트로 되어 있다는 걸 알 수 있습니다. 그리고 왜 훅의 순서가 항상 같아야 하는지도 알 수 있습니다.

// reconciler > ReactFiberHooks.js > mountWorkInProgressHook()
function mountWorkInProgressHook(): Hook {
  // hook 객체에 대해서는 update에서 더욱 자세히 다루게 됩니다.
  const hook: Hook = {
    memoizedState: null, // 마지막 상태값
    queue: null, // 훅이 여러 번 실행될 때 update들을 queue에 연결리스트로 저장한다.
    next: null, // 다음 연결 리스트 노드 포인터

    // 이하 update에서 설명
    baseState: null,
    baseUpdate: null,
  }
  // 맨 처음 실행되는 훅인 경우 연결 리스트의 head로 잡아둠
  if (workInProgressHook === null) {
    firstWorkInProgressHook = workInProgressHook = hook
  } else {
    // 두번 째부터는 연결 리스트에 추가
    workInProgressHook = workInProgressHook.next = hook
  }
  return workInProgressHook
}

혹시나 혼동될까 봐 한 번 더 설명하자면 하나의 컴포넌트에서 여러 훅이 실행될 때는 훅 자체의 next 속성으로 연결 리스트로 저장되고 그 중 하나의 훅이 여러 번 호출될 때는 호출되는 훅 객체의 queue에 그 훅이 만들어놓는 update 정보들이 연결 리스트로 저장됩니다.

function FC() {
  const [a, setA] = useState(0) // aHook
  const [b, setB] = useState(0) // bHook
  setA(_a => _a + 1) // firstUpdate
  setA(_a => _a + 1) // secondUpdate
  setA(_a => _a + 1) // thirdUpdate

  // hook linked list
  // fiber.memoizedState => aHook.next => bHook.next => null

  //  update circular linked list in queue of aHook
  // fiber.memoizedState => aHook.queue.last => thirdUpdate.next
  // firstUpdate.next => secondUpdate.next => thirdUpdate.next => firstUpdate
}

queue는 왜 Circular Linked List인지는 update state에서 설명합니다.

계속해서 mountState로 다시 돌아와서 훅을 mount 할 때 넘겨받은 initialState가 함수이면 바로 실행해서 결괏값을 얻어옵니다. 이 이후 이 함수가 실행되는 일은 없습니다.

// reconciler > ReactFiberHooks.js > mountState()
function mountState(initialState) {
  // const hook = mountWorkInProgressHook();
  if (typeof initialState === 'function') {
    // 생성자 함수일 경우
    initialState = initialState()
  }
  hook.memoizedState = hook.baseState = initialState

2. update를 담을 queue 생성

이제 useState의 상태를 업데이트 하는 함수(이하 setState)를 호출할 때 넘겨주는 인자(이하 Action)를 저장해놓을 queue를 만듭니다. 한 번의 컴포넌트 실행에서 여러 번의 setState가 호출되면 이 queue에 다가 쌓아 놓습니다. 그리고 컴포넌트가 리-렌더링 될 때 queue에 저장되어 있던 Action들을 차례대로 꺼내서 실행해 맨 마지막 결괏값을 state로 반환해줍니다.

// reconciler > ReactFiberHooks.js > mountState()
function mountState(initialState) {
  // hook.memoizedState = hook.baseState = initialState;

  const queue = (hook.queue = {
    last: null, // 마지막 update
    dispatch: null, // queue에 push 해주는 함수

    lastRenderedReducer: basicStateReducer, // return typeof action === 'function' ? action(state) : action;
    lastRenderedState: initialState,
  })

  const dispatch = (queue.dispatch = dispatchAction.bind(
    null,
    currentlyRenderingFiber,
    queue
  ))
  return [hook.memoizedState, dispatch]
}

dispatch 함수 할당을 자세히 보면 bind를 통해서 인자를 잡아둔 상태로 외부에 노출시키고 있습니다. 우리가 사용하고 있는 const [a, setA] = useState(0);가 바로 위 반환하는 배열입니다.

마지막으로 queue에 push 해줄 dispatchAction()를 알아보겠습니다.

3. dispatchAction

이 함수는 queue에 update를 push 함과 동시에 scheduler에게 Work를 스케줄링 시켜주는 함수입니다.
const [a, setA] = useState(0);에서 setA가 dispatchAction()입니다.

reconciler 코드를 보면 항상 render phase가 진행 중일 때 발생한 것인지 아니면 idle 상태에서 발생한 것인지에 따라 로직이 나뉩니다.
두 케이스 모두 처리해야 할 방법이 다르고 최적화 방식도 조금씩 다르므로 기능이 궁금해서 함수를 까보면 이 이유 때문에 코드의 길이가 상당히 긴 경우가 많습니다.
그래서 현재 분석하고 있는 코드가 어떤 상황에서 사용되는지 확실히 알고 가야 이해하기가 편합니다.

render phase update란?
컴포넌트가 렌더링 되고 있는 상황에서 추가적으로 업데이트가 발생할 경우를 말합니다.
다음 코드에서 버튼을 클릭했을 때 컴포넌트는 a가 1인 상태로 실행되는데 setA(2) 때문에 추가적인 업데이트가 발생한 경우입니다.

function FC() {
  const [a, setA] = useState(0)

renderWithHooks()를 통해 컴포넌트가 실행되는 상황은 VDOM 재조정 작업 중 일때 발생합니다. 이는 업데이트가 발생하면 진행되는 작업으로 currentlyRenderingFiber가 비어있지 않다는 건 renderWithHooks()가 호출되었다는 뜻입니다. 즉 render phase가 진행 중인 상황입니다.

이때 10번 라인 dispatchAction()이 bind로 잡아 두었던 fiber와 현재 작업 중인 currentlyRenderingFiber 가 같다는 건 현재 컴포넌트는 render phase에서 작업중인 컴포넌트이며 동시에 업데이트가 발생한 상황임을 나타내는 조건입니다.

한가지 더 alternate도 비교를 하는데(11번 라인) VDOM 트리는 두 개가 있습니다. current, workInProgress.
하지만 우리는 훅이 mount 될 때 작업 중이었던 fiber를 bind로 고정시켜 놨습니다.
거기에 current와 workInProgress는 고정이 아닌 commit이 되면 서로 교체가 되기 때문에 현재 작업 중인 currentlyRenderingFiber가 두 개의 트리 중 어느 트리인지 알 수가 없습니다.
그래서 fiber와 alternate를 모두 비교를 해야 현재 컴포넌트가 작업 중인지 확인할 수 있습니다.

render phase update 로직은 idle update를 먼저 보고 난 후에 확인해 보도록 하겠습니다.

4. expirationTime

idle update를 들어가기에 앞서 먼저 expirationTime 부터 알고 가야 합니다.
expirationTime에 대해서는 깊이 들어가지는 않지만 리액트 시스템에 전반적으로 사용되므로 아주 기본만 알고 넘어가겠습니다. 가장 큰 expirationTime은 32-bit 시스템인 v8에서 가장 큰 정수인 부호 있는 31bit 값입니다. 이 값을 Sync로 다룹니다. 그리고 legacy mode에서는 모든 게 Sync로 처리된다고 생각하시면 됩니다.
여기서 나오는 나머지 부분들은 concurrent mode에서 사용됩니다.
그리고 이름이 expirationTime이라서 그렇지 작업의 유효 시간을 나타내는 것으로 쓰이지는 않습니다..

// reconciler > ReactFiberExpirationTime.js
export const NoWork = 0
export const Never = 1
export const Idle = 2
export const Sync = MAX_SIGNED_31_BIT_INT // 1073741823
export const Batched = Sync - 1

const MAGIC_NUMBER_OFFSET = Batched - 1

현재 시간을 가지고 올 때 시간의 형식은 Date.now()가 아닌 performance.now() 입니다.
expirationTime을 구하는 계산식은 MAX_SIGNED_31_BIT_INT - now() 입니다.
now가 오른쪽 피 연산자에 있다는 건 나중에 발생한 작업일수록 더 작은 expirationTime을 가지게 된다는 뜻입니다. 하지만 performance.now()로 현재 시간을 표현하기 때문에 그럴 일은 없겠지만 이론상으로는 발생 시간이 0이 나올 수도 있습니다.
그렇게 되면 Sync나 Batched와 시간이 겹칠 수 있어서 기존 상수들과 겹치지 않기 위해 offset 값이 필요한데 그게 바로 MAGIC_NUMBER_OFFSET입니다. 위 이유 때문에 계산식을 다시 작성하자면 MAGIC_NUMBER_OFFSET - now()가 됩니다.
그리고 반대로 expirationTime에서 발생시간을 구하는 계산식은 offset - expirationTime expirationTime 입니다.
expirationTime와는 반대로 큰 숫자가 더 나중에 발생한 작업을 나타냅니다.

// reconciler > ReactFiberExpirationTime.js
// 생략..
///export const Batched = Sync - 1;

const UNIT_SIZE = 10

// 1 unit of expiration time represents 10ms.
export function msToExpirationTime(ms: number): ExpirationTime {
  return MAGIC_NUMBER_OFFSET - ((ms / UNIT_SIZE) | 0)
}

export function expirationTimeToMs(expirationTime: ExpirationTime): number {
  return (MAGIC_NUMBER_OFFSET - expirationTime) * UNIT_SIZE
}

다시 간략하게 정리하자면 fiber의 expirationTime이 NoWork이면 작업 중인게 없다는 뜻이고 Sync는 동기적으로 작업이 처리될 것이며 expirationTime을 대소로 비교하는 조건문이 있을 때는 큰 숫자가 더 옛날에 먼저 발생한 작업이다. 라고만 생각하고 넘어가시면 분석할 때 큰 무리 없이 이해할 수 있을 것입니다.
시간과 관련된 부분은 concurrent mode가 일반적으로 사용이 되면 그때 깊이 있게 알아보도록 하겠습니다.

다시 훅의 업데이트를 dispatch 해주는 dispatchAction로 돌아오겠습니다. expirationTime을 간략하게 알아보았기 때문에 현재 시간으로 fiber의 expirationTime을 구하는 부분은 생략하도록 하겠습니다. 업데이트에 대한 시간을 구한 후 이제 해야할 일은 업데이트 정보를 담을 update객체를 만드는 것입니다.
그리고 이 Update를 queue에 push 해야 합니다.

7. 정리

이제 여러분은 다음과 같은 질문에 대답할 수 있어야 합니다.

  1. setState를 한번에 여러 번 실행하면 매 실행마다 렌더링 되나요?
  2. 클릭을 통해 컴포넌트 상태를 업데이트시켰습니다. 그리고 변경된 상태를 기준으로 setState를 호출했어요. 그러면 클릭을 통한 상태 업데이트 때 한번, 추가적인 setState 때 또 한 번. 이렇게 매번 렌더링 되나요?

이런 질문이 들어오면 일단 가장 먼저 질문자에게 해주어야 할 부분은 렌더링 단어의 의미를 명확하게 인식시키는 겁니다. 이 부분만 제대로 이해해도 개발자의 입장에서만 바라보던 것들을 리액트의 입장에서 바라볼 수 있도록 시야를 트여줍니다.

이제 하나씩 답변을 해보겠습니다.

  1. 복수개의 훅 실행이 현재 state와 같다면 아무 일도 일어나지 않는다. 그게 아니면 한데 묶어서 다음 프레임 때 한 번만 컴포넌트가 재실행된다. 화면에 그려지는 것 또한 한 번이다.(스케줄링도 한번)
  2. 클릭을 통한 업데이트로 컴포넌트가 재실행 된다. 그리고 그 재실행에서 발생하는 추가적인 setState는 1번처럼 묶어서 한 번만 실행한다.
    여기서 업데이트가 발생하지 않을 때까지 계속해서 함수를 재실행한다. 결국, 최소 2번 최대 25번(RERENDERLIMIT = 25) 컴포넌트가 재 실행될 수 있다. 그리고 화면은 1번과 마찬가지로 한 번만 그려진다.
    이 사이클은 render phase임을 유의해라. commit phase가 아니면 단지 in-memory 객체만 끄적인 것이지 아무 변화도 일어나지 않는다.

여기까지 이해가 가시나요? 그러면 다행이지만 그렇지 않다면 가장 이해하기 쉬운 방법은 위 내용을 토대로 직접 코드를 작성해서 디버깅을 해보는 것입니다.

다음은 setState를 통해 queue에 저장해두었던 update들을 훅이 어떻게 컴포넌트에 적용시켜 실행하는지 알아보도록 하겠습니다.
